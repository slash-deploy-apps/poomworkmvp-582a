## ADDED Requirements

### Requirement: 채팅방 내 금액 제안 메시지

시스템은 전문가가 채팅방에서 금액과 예상 기간을 제안하는 메시지를 별도 카드 UI로 표시해야 한다.

#### Scenario: 제안 메시지 전송

- **WHEN** 전문가가 채팅 입력창의 "금액 제안" 버튼을 클릭하고 금액/기간을 입력하여 전송하면
- **THEN** 시스템은 `messages` 테이블에 `type='proposal'`인 레코드를 저장하고, 채팅 내역에 제안 카드를 렌더링한다

#### Scenario: 제안 메시지 렌더링

- **WHEN** 사용자가 채팅방을 열었을 때 `type='proposal'`인 메시지가 있으면
- **THEN** 시스템은 금액, 기간, 상태(제안중/수락/거부)를 포함한 카드 UI를 표시한다

### Requirement: 제안 수락 및 거부

시스템은 의뢰자가 제안을 수락하거나 거부할 수 있게 해야 한다.

#### Scenario: 제안 수락

- **WHEN** 의뢰자가 제안 카드의 "수락" 버튼을 클릭하면
- **THEN** 시스템은 제안 상태를 `accepted`로 변경하고 계약 동의 페이지로 리다이렉트한다

#### Scenario: 제안 거부

- **WHEN** 의뢰자가 제안 카드의 "거부" 버튼을 클릭하면
- **THEN** 시스템은 제안 상태를 `rejected`로 변경하고, 전문가에게 거부 알림 메시지를 별도 `type='system'` 메시지로 전송한다

#### Scenario: 재제안

- **WHEN** 전문가가 거부된 제안에 대해 새로운 금액으로 재제안하면
- **THEN** 시스템은 새로운 `type='proposal'` 메시지를 생성하고 이전 제안은 무효화 처리한다

### Requirement: 제안과 지원서 연결

시스템은 제안 메시지가 특정 지원서와 연결되어 있음을 보장해야 한다.

#### Scenario: 지원서 기반 제안

- **WHEN** 전문가가 금액을 제안하면
- **THEN** 시스템은 해당 메시지의 `jobId`를 통해 관련 지원서를 찾아 `applicationId`를 메타데이터에 저장한다
