## Context

현재 poomwork 플랫폼은 일감(`jobs`) 등록 → 지원서(`jobApplications`) 제출 → 의뢰자 수락 및 NicePay 직접 결제 플로우를 가지고 있습니다. 그러나 이 플로우는 에스크로 보호가 없어 양측 모두 거래 리스크를 부담합니다. 기존 `payments` 테이블에는 `escrowReleasedAt` 필드가 있지만 실제 에스크로 로직은 미구현 상태입니다.

이 디자인은 채팅 기반 금액 제안 → 계약 동의 → 에스크로 결제 → 결과물 전달 → 컨펌/거부 → 에스크로 해제의 완전한 플로우를 구현합니다.

## Goals / Non-Goals

**Goals:**

- 전문가가 채팅방에서 의뢰자에게 금액/기간을 제안할 수 있음
- 의뢰자가 제안을 수락하면 계약 동의 페이지로 이동
- 양측이 각자 동의 체크 후 확인 → 3초 폴리 태 리로 실시간 동의 상태 공유
- 동의 완료 시 의뢰자에게 NicePay 결제창 실행
- 결제 완료 시 `payments` 레코드가 `escrow` 상태로 생성되고 전문가에게 "일 시작" 안내
- 전문가가 결과물(UploadThing 파일 + 텍스트/링크)을 업로드하여 의뢰자에게 전달
- 의뢰자가 컨펌(수락) 또는 거부(재작업 요청) 가능
- 컨펌 시 `payments` 상태를 `escrow_released`로 변경하고 전문가에게 지급 완료 안내
- 7일 미응답 시 자동 에스크로 해제 (SQLite 백그라운드 로직 또는 매 요청 시 체크)
- 지원서 상태를 계약 단계별로 추적

**Non-Goals:**

- 실제 은행 계좌 이체 API 연동 (가상 구현)
- 실시간 WebSocket (3초 폴리 태 리로 대체)
- 별도의 관리자 에스크로 중재 UI
- 다중 결제 수단 지원 (NicePay 카드 결제만)
- 계약서 PDF 생성 또는 전자서명 법적 효력

## Decisions

### 1. 계약 엔터티를 별도 `contracts` 테이블로 분리

- **선택**: `contracts` 테이블 신규 생성 (`jobApplications`와 1:1 또는 1:N)
- **근거**: 계약은 지원서와 별개의 라이프사이클을 가짐. 한 지원서에 대해 여러 번 제안→거부→재제안이 가능하므로 `contracts`는 지원서를 참조하되 독립적인 상태 머신을 가짐.
- **대안**: `jobApplications`에 계약 필드를 추가 → 거부 시 필드 오염 및 이력 관리 어려움

### 2. 3초 폴리 태 리(polling)로 실시간 동의 표시

- **선택**: `useEffect` + `setInterval(3000)`로 `/api/contracts/:id/status`를 폴리 태 리
- **근거**: WebSocket 도입 시 인프라 복잡도 증가. React Router의 서버 렌더링 모델과 폴리 태 리가 자연스럽게 어울림. 3초면 사용자가 실시간처럼 느낌.
- **대안**: WebSocket(SSE) → 서버 사이드 상태 관리 및 연결 관리 부담

### 3. 제안 메시지를 `messages` 테이블에 통합 저장

- **선택**: `messages`에 `type` 필드를 추가(`'text' | 'proposal' | 'system'`)하고, `proposal` 타입일 때 `metadata` JSON에 금액/기간/상태 저장
- **근거**: 채팅 내역의 자연스러운 일부로 제안을 표시하기 위해. 별도 테이블은 조인 복잡도 증가.
- **대안**: 별도 `proposals` 테이블 + 메시지 참조 → UI 조합 로직 복잡

### 4. 결과물 전달은 UploadThing + 텍스트 필드

- **선택**: 파일은 UploadThing API 사용, 설명은 `contracts.deliverableText` 컬럼에 저장
- **근거**: 프로젝트가 이미 UploadThing을 사용 중이며, 별도 파일 스토리지 인프라 불필요.
- **대안**: Base64 DB 저장 → 용량 및 성능 문제

### 5. 7일 자동 해제는 매 API 요청 시 체크

- **선택**: `deliveredAt` + 7일을 계산하여 매 컨트랙트 상태 조회 API에서 현재 시간과 비교
- **근거**: SQLite 기반 프로젝트에서 cron job이나 백그라운드 worker 추가는 오버엔지니어링.
- **대안**: node-cron 라이브러리 → 별도 프로세스 관리 필요

## Risks / Trade-offs

| Risk                                                            | Mitigation                                                                                                                            |
| --------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| 폴리 태 리로 인한 서버 부하 증가                                | 계약 페이지 진입 시에만 폴리 태 리 실행, 페이지 이탈 시 클리어. 캐싱 불필요 (저트래픽)                                                |
| 의뢰자가 7일 내 컨펌 없이도 에스크로 해제를 원하지 않을 수 있음 | 7일은 충분한 기간이며, 의뢰자가 컨펌/거부 버튼을 명시적으로 누를 수 있음. 거부 시 타이머 리셋                                         |
| 전문가가 결과물을 업로드하지 않고 일감 완료를 주장할 수 있음    | 결과물 업로드 없이는 `delivered` 상태로 전환 불가. 의뢰자가 결과물 미수령 시 거부 가능                                                |
| NicePay 에스크로 API와 연동 시 복잡도                           | 1차적으로는 NicePay 일반 결제를 그대로 사용하고, 에스크로 상태는 낮은 시스템(DB)에서만 관리. 추후 NicePay 에스크로 API 연동 가능      |
| 동시 동의 체크 시 레이스 컨디션                                 | DB 트랜잭션으로 `workerAgreed`와 `clientAgreed` 업데이트 시 원자성 보장. 양측이 동시에 확인을 눌러도 최종 상태는 `both_agreed`로 수렴 |

## Migration Plan

1. **DB 마이그레이션**: `contracts` 테이블 생성, `messages`에 `type`/`metadata` 추가, `jobApplications` 상태 enum 확장
2. **API 개발**: 계약 CRUD, 동의 API, 결과물 업로드 API, 컨펌/거부 API
3. **UI 개발**: 채팅 제안 카드, 계약 동의 페이지, 결과물 전달/컨펌 페이지
4. **통합 테스트**: NicePay 결제 흐름과 에스크로 상태 연동 테스트
5. **롤백**: 마이그레이션 되돌리기(drizzle drop) 시 기존 데이터 유지

## Open Questions

- NicePay 에스크로 API를 향후 연동할 때, 현재 DB 기반 에스크로 상태와의 동기화 전략
- 의뢰자가 결과물 거부 시 재작업 요청 횟수 제한 필요 여부 (현재는 무제한 허용)
