## ADDED Requirements

### Requirement: Render TossPayments Payment Widget
The system SHALL render the TossPayments v2 Payment Widget inside the course purchase dialog when the user clicks the purchase button for a paid course.

#### Scenario: Widget renders for paid course
- **WHEN** a logged-in user clicks "수강 신청하기" on a course with price > 0
- **THEN** the payment dialog opens and the TossPayments Payment Widget is rendered inside it
- **AND** the widget displays the correct amount (course price in KRW)
- **AND** the widget allows selection of payment methods (card, KakaoPay, NaverPay, etc.)

### Requirement: Request payment via widget
The system SHALL initiate payment through the widget when the user confirms payment.

#### Scenario: Payment request succeeds
- **WHEN** the user selects a payment method and clicks the payment confirmation button in the widget
- **THEN** the system calls `widgets.requestPayment()` with the following parameters:
  - `orderId`: a unique UUID generated by the client
  - `orderName`: the course title
  - `successUrl`: `{window.location.origin}/payment/success?orderId={orderId}&courseId={courseId}`
  - `failUrl`: `{window.location.origin}/payment/fail?orderId={orderId}&courseId={courseId}`
  - `customerKey`: the logged-in user's ID
- **AND** the user is redirected to the TossPayments payment flow

#### Scenario: Free course enrollment
- **WHEN** the user clicks "묶료로 시작하기" on a course with price = 0
- **THEN** no payment widget is shown and no TossPayments interaction occurs
- **AND** the system immediately creates an `enrollments` record with `userId`, `courseId`, `progress = 0`
- **AND** the system increments the course `enrollmentCount` by 1
- **AND** the user is redirected to `/courses/{courseId}/learn`

#### Scenario: Free course enrollment with existing enrollment
- **WHEN** the user clicks "묶료로 시작하기" on a free course they are already enrolled in
- **THEN** the system detects the existing enrollment
- **AND** redirects the user directly to `/courses/{courseId}/learn` without creating a duplicate enrollment

### Requirement: Payment preparation endpoint
The system SHALL provide a server endpoint to prepare a payment before rendering the widget.

#### Scenario: Prepare payment record
- **WHEN** the client sends a POST request to `/api/payment/prepare` with `{ courseId, orderId }`
- **THEN** the server creates a `payments` record with:
  - `payerId`: current user's ID
  - `payeeId`: course instructor's ID
  - `amount`: course price
  - `type`: 'course_purchase'
  - `status`: 'PENDING'
  - `referenceId`: course ID
  - `orderId`: the provided orderId
- **AND** returns `{ success: true }`
- **AND** if the user already has a PENDING payment for this course, returns the existing orderId
